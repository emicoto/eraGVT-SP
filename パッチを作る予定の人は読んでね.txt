### パッチを取りこんでもらえるか？採用基準は？

erGVT別人改造版では「パッチを作る」という労力に対して敬意を払い、
基本的にうｐされたパッチは全て取り込む方針を取っています。
ユーザが個人で改造する場合は更新ごとにやり直す必要があるので、
その手間を軽減する意味でもパッチはなるべく取り込むことにしています。


ただし、以下の項目に当てはまる場合はパッチの取り込みをお断りさせて頂く場合があります。
・大量のバグの発生が予想される、または将来的に大量の修正箇所が発生すると予想される場合
・ゲーム性の面で問題があり、かつバランス調整で問題を解決できない場合


もし不安なら開発スレにて「○○という機能のパッチを作る予定なんですが問題ありませんか」と
事前に相談して頂ければ回答させて頂きます。




###　守るべきルール

このセクションではeraGVT別人改造版でのコーディングにおける最低限のルールについて述べます。
基本的に各パッチ作者の裁量で好きな書式でコードを記述して構いませんが、
修正作業の労力軽減のために以下の点については留意しておいてください。



1.インデント(タブスペース)管理を徹底する

　erabasic言語ではIF~ENDIF、SELECTCASE~ENDSELECT、WHILE~WENDなどのループ処理を行う際に、
　ループ階層を分かりやすくするためにタブスペースで字下げを行う共通のルールが存在します。
　逆に、ループ階層分けをする以外の用途でインデントを行わないでください。

　これは可読性を維持するためのルールです。



2.不等号(<,<=,>,>=,==)の省略はなるべく避ける

　IF TALENT:999　のような不等号を省略した記述はなるべく避けてください。

　erabasic言語では不等号を省略した場合は != 0を意味します。
　例えば　IF TALENT:999　と記述した場合は IF TALENT:999 != 0　と同じ意味を持ちます。
　負の数だった場合でもtrue判定されてしまうので、
　正の値を想定している場合は　IF TALENT:999 > 0 と記述するようにしてください。
　正負両方の値を想定している場合は　IF TALENT:999 != 0　と記述し、
　記述ミスではないことをはっきり示すようにしてください。

　これは将来の改変によって不具合が発生したり、
　大量の修正箇所が発生してしまうのを未然に防ぐためのルールです。



3.一文字変数(A〜Z)は使用しない

　erabasic言語ではA:0、A:1、A:2といった具合にアルファベット一文字で表現される変数が使用可能ですが、
　この一文字変数を使用することは避けてください。
　アルファベット一文字という性質上GREP検索で特定することが困難であり、
　将来的にバグ修正の手間を増大させる可能性があるからです。

　これはコードの健全性を維持するためのルールです。
